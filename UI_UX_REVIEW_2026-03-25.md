## 반영 상태

- 본 문서의 확정 이슈 6건과 추가 제안 4건은 2026-03-25 작업에서 모두 반영 완료
- 반영 범위:
  - 검색 busy state/위험 액션 차단
  - 검색 가능 상태 gating 정리
  - 인라인 검색 진행 카드 + 취소
  - 무결과 필터 요약 + `필터 초기화`

  - 검색 히스토리 확장 저장 및 조건 복원 즉시 재검색

## 확정 이슈

### 1. [높음] 문서 처리/모델 다운로드 중에도 배경 조작이 가능해서 상태 충돌 위험이 있다

- 근거 코드
  - `ProgressDialog`는 `QDialog`가 아니라 `QFrame`이다: `regfinder/ui_components.py:283`
  - 문서 처리 시 `show()`만 호출하고 모달 처리나 배경 비활성화가 없다: `regfinder/main_window.py:365-380`
  - 모델 즉시 변경은 별도 guard 없이 런타임 상태를 바로 초기화한다: `regfinder/main_window.py:311-325`
  - 캐시 삭제도 진행 중 작업 확인 없이 바로 수행된다: `regfinder/main_window.py:771-776`
- 영향
  - 인덱싱 중 설정 탭으로 이동해 `모델 즉시 변경`이나 `캐시 삭제`를 누를 수 있다.
  - 이 경우 백그라운드 워커가 쓰고 있는 `qa` 상태와 UI가 동시에 리셋되어 결과가 뒤섞이거나, 작업 완료 후 화면이 예기치 않게 덮어써질 수 있다.
- 판단
  - 단순 UX 문제가 아니라 상태 일관성 문제다. 가장 먼저 막아야 한다.
- 권장 대응
  - 진행 다이얼로그를 `QDialog` 기반 모달로 바꾸거나, 작업 중에는 탭/설정/캐시/모델 변경 액션을 일괄 비활성화한다.
  - 최소한 `_reload_model()` / `_clear_cache()` / `_download_all_models()` 진입 시 `docs`, `model`, `download`, `search` 워커 상태를 확인해야 한다.

### 2. [높음] 파일/북마크 테이블의 숫자 정렬이 실제 숫자 기준으로 동작하지 않는다

- 근거 코드
  - 파일 탭은 `크기`, `청크`에 대해 숫자를 `UserRole + 1`에만 넣고, 표시값은 문자열로 넣는다: `regfinder/main_window.py:466-472`
  - 북마크 탭도 `랭킹 점수`를 문자열 `QTableWidgetItem`으로만 넣는다: `regfinder/main_window_mixins.py:206-209`
  - 두 테이블 모두 정렬을 활성화한다: `regfinder/main_window.py:479`, `regfinder/main_window_mixins.py:140`
- 영향
  - 사용자는 `크기`, `청크`, `랭킹 점수`로 정렬한다고 생각하지만 실제로는 문자열 비교가 섞여 신뢰할 수 없는 순서가 나온다.
  - 예: `"10.0"`, `"2.0"`, `"9.0"` 정렬 시 문자열 기준으로 `"10.0", "2.0", "9.0"` 순서가 재현된다.
- 판단
  - "정렬된다"는 UI 약속이 깨지는 기능 이슈다.
- 권장 대응
  - 숫자 정렬용 `QTableWidgetItem` 서브클래스를 두거나, `DisplayRole` 대신 정렬 가능한 숫자 역할을 사용하도록 바꾼다.

### 3. [중간] 모델만 로드되면 검색창이 활성화되어, 실제로는 검색 불가한 상태를 검색 가능으로 보이게 만든다

- 근거 코드
  - 모델 로드 성공 직후 검색 입력과 검색 버튼을 바로 켠다: `regfinder/main_window.py:248-253`
  - 인덱스 리셋 후에도 `model_ready`만 보면 검색창을 다시 켠다: `regfinder/main_window.py:169-173`
  - 그러나 실제 검색 진입 시 인덱스가 없으면 "문서를 먼저 로드하세요" 경고를 띄운다: `regfinder/main_window.py:503-504`
  - 캐시 삭제 후에도 같은 흐름이 유지된다: `regfinder/main_window.py:771-776`
- 영향
  - 사용자는 "검색창이 활성화 = 검색 준비 완료"로 인식하기 쉽다.
  - 현재 구현은 모델 로드 후나 캐시 삭제 후에 에러를 먼저 경험하게 만든다.
- 판단
  - 기능은 맞지만, 첫인상과 학습 비용 면에서 손해가 큰 흐름이다.
- 권장 대응
  - 검색창 활성 조건을 `현재 검색 모드 존재 여부`로 바꾸는 편이 자연스럽다.
  - 또는 검색창은 활성화하되 입력창 내부/헤더 상태에 "먼저 폴더를 로드하세요"를 명시하고 검색 버튼은 비활성화하는 방식이 더 낫다.

### 4. [중간] 검색 중 필터/정렬/결과 개수/히스토리 버튼이 계속 살아 있어, 화면 상태와 실제 검색 조건이 엇갈릴 수 있다

- 근거 코드
  - 검색 중 비활성화되는 것은 `search_input`, `search_btn`뿐이다: `regfinder/main_window.py:81-83`, `regfinder/main_window.py:517`
  - 같은 패널의 `filename_filter_input`, `path_filter_input`, `ext_filter_combo`, `sort_combo`, `history_btn`, `k_spin`은 별도로 잠그지 않는다: `regfinder/main_window_ui_mixin.py:187-230`
  - 검색 결과 헤더에는 실제로 적용된 필터/정렬 정보가 표시되지 않는다: `regfinder/main_window.py:572-603`
- 영향
  - 사용자가 검색 직후 필터나 정렬을 바꾸면, 화면에 보이는 컨트롤 상태와 방금 나온 결과 조건이 달라질 수 있다.
  - "왜 이 정렬로 안 나왔지?" 같은 혼란이 생기기 쉽다.
- 판단
  - 데이터 무결성 문제는 아니지만, 결과 해석을 흔드는 전형적인 UX 결함이다.
- 권장 대응
  - 검색 중에는 해당 패널 전체를 잠그거나, 결과 헤더에 `적용 필터/정렬`을 명시해야 한다.

### 5. [중간] "결과 폰트 크기" 설정이 현재 보이는 결과 카드에 즉시 반영되지 않는다

- 근거 코드
  - 폰트 슬라이더 변경 시 수행하는 작업은 상태 저장과 라벨 갱신뿐이다: `regfinder/main_window_mixins.py:41-45`
  - 결과 카드의 본문 폰트는 생성 시점의 `font_size`로만 지정된다: `regfinder/ui_components.py:208`
- 영향
  - 사용자는 설정 탭에서 폰트 크기를 바꿨는데도, 현재 열린 검색 결과가 그대로여서 설정이 먹지 않는다고 느낀다.
  - 특히 접근성 목적의 글자 확대에서 체감이 크다.
- 판단
  - 설정 의미와 실제 반응이 어긋나는 문제다.
- 권장 대응
  - 현재 렌더된 `ResultCard`의 `QTextEdit` 폰트를 순회 갱신하거나, 결과 목록을 현재 데이터로 다시 렌더링해야 한다.

### 6. [낮음] 결과 카드의 파일 링크 툴팁과 실제 동작이 다르다

- 근거 코드
  - 툴팁 문구는 `더블클릭으로 파일 열기`: `regfinder/ui_components.py:71`
  - 실제 동작은 좌클릭 한 번으로 즉시 열기: `regfinder/ui_components.py:74-77`
- 영향
  - 사용자는 더블클릭을 기대하지만 단일 클릭으로 파일이 열려 의도치 않은 실행으로 느낄 수 있다.
- 판단
  - 작은 문제지만 신뢰도에 영향을 준다.
- 권장 대응
  - 툴팁을 "클릭하여 파일 열기"로 바꾸거나, 실제 더블클릭 동작으로 맞춰야 한다.

## 추가 제안

### 1. 검색에도 진행 상태와 취소 UX를 추가하는 편이 좋다

- 근거 코드
  - `SearchThread`는 `finished`만 있고 중간 진행 신호가 없다: `regfinder/workers.py:261-284`
  - 현재 검색 중 화면은 정적 라벨 `"🔍 검색 중..."` 한 줄이다: `regfinder/main_window.py:517-520`
- 제안
  - 대용량 인덱스에서는 검색도 체감 대기 시간이 생길 수 있으므로, 최소한 취소 버튼과 elapsed time 정도는 표시하는 편이 낫다.

### 2. "검색 결과 없음" 상태에 현재 필터 조건을 함께 보여줘야 한다

- 근거 코드
  - 빈 결과 메시지는 항상 `"다른 검색어로 시도해보세요."`로 고정된다: `regfinder/main_window.py:686-690`
  - 하지만 현재 앱은 확장자/파일명/경로 필터를 적극 지원한다: `regfinder/main_window_ui_mixin.py:187-217`
- 제안
  - `현재 필터: PDF / 파일명=휴가 / 경로=인사` 같은 요약과 함께 "필터 초기화" 액션을 주면 원인 파악이 빨라진다.

### 3. 검색 히스토리에 쿼리만이 아니라 필터/정렬 조건도 같이 저장하는 편이 좋다

- 근거 코드
  - 히스토리는 `q` 한 필드만 저장한다: `regfinder/ui_components.py:43-45`, `regfinder/ui_components.py:52-59`
- 제안
  - 현재 구조에서는 "같은 검색어인데 왜 지난번과 다른 결과가 나오지?"가 생기기 쉽다.
  - 검색어 재실행 시 함께 복원할 필터/정렬/검색 모드를 저장하면 분석성과 재현성이 크게 올라간다.

### 4. 검색 결과 카드에서 본문 전체 보기/근거 청크 이동이 있으면 탐색 효율이 더 좋아진다

- 근거 코드
  - 결과는 파일 단위로 묶이지만, 카드에는 대표 청크와 근거 청크 개수만 나온다: `regfinder/ui_components.py:197-201`
  - 본문 박스는 높이가 제한돼 있고 현재는 별도 확장 액션이 없다: `regfinder/ui_components.py:206-210`
- 제안
  - "전체 보기", "다음 근거 청크", "파일 경로 복사" 정도만 추가해도 실제 검토 속도가 많이 좋아질 가능성이 높다.

## 우선순위 제안

1. 진행 중 배경 조작 차단
2. 숫자 정렬 수정
3. 검색 가능 상태 기준 정리
4. 검색 중 컨트롤 잠금/조건 표시
5. 폰트 크기 즉시 반영
6. 나머지 UX 개선

## 간단 재현 메모

- 숫자 정렬 재현: 오프스크린 `QTableWidget`에서 `"10.0"`, `"2.0"`, `"9.0"`가 문자열 순서로 정렬되는 것을 확인했다.
- 폰트 즉시 반영 재현: 결과 카드 생성 후 `_on_font_size_changed(24)`를 호출해도 기존 `QTextEdit` 폰트 크기가 그대로 유지되는 것을 확인했다.
- 검색 중 컨트롤 상태 재현: 검색 스레드를 지연시킨 상태에서 `filename/path/ext/sort/history/k` 컨트롤이 계속 활성인 것을 확인했다.
- 문서 처리 중 상태 충돌 재현: `docs` 워커가 실행 중인 상태에서도 `_reload_model()`이 그대로 실행되는 것을 확인했다.
